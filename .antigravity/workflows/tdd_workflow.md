# TDD (Test-Driven Development) Workflow

## Purpose
Standardize the Test-Driven Development process across all agents to ensure high-quality, test-covered code.

## Participants
- **Orchestrator**: Coordinates workflow, validates acceptance
- **PythonBackendDeveloper**: Implements production code
- **QADeveloper**: Writes tests (may collaborate or lead testing)

## Workflow Steps

### Phase 1: Requirement Analysis
1. **Orchestrator** receives a feature request or bug fix specification.
2. **Orchestrator** uses Specs Driven Development skill to parse requirements into:
   - Clear acceptance criteria
   - API contract (if applicable)
   - Domain model changes
3. **Orchestrator** decomposes feature into atomic development tasks.
4. **Orchestrator** assigns first task (usually starting with tests) to appropriate agent(s).

### Phase 2: Test First (Red)
1. **QADeveloper** (or **PythonBackendDeveloper** in TDD mode) writes a failing test that captures one piece of functionality.
   - Test should be minimal and focused.
   - Test should fail for the expected reason (not compilation errors).
2. Test is committed/run to confirm it fails (Red state).
3. **Orchestrator** verifies test failure matches expectation.

### Phase 3: Implement Minimal Code (Green)
1. **PythonBackendDeveloper** writes the minimal production code to make the test pass.
   - No extra functionality beyond what the test requires.
   - Can be inelegant or duplicative initially.
2. Code is run against the test to confirm it passes (Green state).
3. **Orchestrator** verifies test passes.

### Phase 4: Refactor
1. **PythonBackendDeveloper** refactors the code while keeping tests green.
   - Improve design, remove duplication, apply patterns.
   - Ensure code follows project standards (formatting, typing, etc.).
2. **QADeveloper** may also refactor tests for clarity and maintainability.
3. Run test suite to ensure no regression.

### Phase 5: Repeat
1. Repeat steps 2-4 for the next piece of functionality.
2. Continue until all acceptance criteria are met.

### Phase 6: Integration
1. Once all unit tests pass, **QADeveloper** writes integration tests for the feature.
2. **PythonBackendDeveloper** fixes any integration issues.
3. **Orchestrator** runs full test suite (unit + integration) to validate the feature.

### Phase 7: Validation
1. **Orchestrator** checks that all acceptance criteria are satisfied.
2. **Orchestrator** ensures test coverage meets project standards (≥90%).
3. **Orchestrator** marks task as complete and moves to next task or feature.

## Collaboration Patterns

### Pattern A: QADeveloper Leads Testing
- **QADeveloper** writes all tests (unit and integration).
- **PythonBackendDeveloper** implements code to pass tests.
- Best for complex logic where test design requires specialized QA expertise.

### Pattern B: PythonBackendDeveloper Self-TDD
- **PythonBackendDeveloper** writes both tests and code following TDD.
- **QADeveloper** reviews tests for completeness and edge cases.
- Faster for straightforward features; maintains developer ownership.

### Pattern C: Paired TDD
- **PythonBackendDeveloper** and **QADeveloper** collaborate in real-time.
- One writes test, other writes code, then switch roles.
- Highest quality but most resource-intensive.

## Artifacts

### 1. Test Files
- Unit tests in `tests/unit/`
- Integration tests in `tests/integration/`
- Each test file corresponds to a production file.

### 2. Production Code
- Implemented in appropriate layer (controller, service, repository).
- Follows project architecture and conventions.

### 3. Coverage Reports
- Generated by `pytest-cov` in CI pipeline.
- Must maintain or improve overall coverage.

### 4. Acceptance Checklist
- Markdown checklist of acceptance criteria.
- Each item linked to specific tests.

## Quality Gates

### 1. Test Failure Verification
- Each new test must fail before implementation.
- Prevents false positives (tests that pass without implementation).

### 2. Minimal Implementation
- Code should be just enough to pass tests.
- Prevents over-engineering and scope creep.

### 3. Refactoring Safety
- All tests must pass after refactoring.
- Use test suite as safety net.

### 4. Coverage Threshold
- New code must meet coverage target.
- Critical paths require 100% coverage.

## Example: Implementing User Registration

### Step 1: Acceptance Criteria (from spec)
- [ ] POST /users with valid email/password returns 201
- [ ] Email must be unique
- [ ] Password must be at least 8 characters
- [ ] Response includes user ID and email

### Step 2: Task Decomposition
1. Create User DTOs with validation
2. Implement UserRepository.create
3. Implement UserService.register (with password hashing)
4. Implement UserController.post_users
5. Write unit tests for each layer
6. Write integration test for endpoint

### Step 3: TDD Iteration for DTOs
1. **Red**: Write test for UserCreateDTO validation (test valid/invalid email, password length)
2. **Green**: Implement Pydantic model with EmailStr and min_length
3. **Refactor**: Extract email validation logic if repeated

### Step 4: Continue through layers
- Repeat for repository, service, controller.
- Each layer tested in isolation first, then integration.

### Step 5: Final Validation
- Run full test suite
- Verify all acceptance criteria checked
- Ensure coverage target met

## Tools
- **Testing**: pytest, pytest-cov, pytest-mock
- **CI/CD**: GitHub Actions (runs tests on each push)
- **Monitoring**: Coverage reports, test duration tracking

## Exit Criteria
- All acceptance criteria satisfied
- Test suite passes (unit + integration)
- Coverage meets or exceeds target
- Code reviewed and merged via standard process
